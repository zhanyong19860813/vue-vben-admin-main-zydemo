### 2) 组织/人员 API 的取数来源
- 部门 API 使用：`CommonDB.GetDeptInfo(...)`，底层视图 `v_t_base_department`
- 人员 API 使用：`CommonDB.GetUserAllInfo(...)`，底层视图 `v_t_employee_info`

### 3) 同步服务（PersonalInfoSyncService）调用逻辑
- 调度频率：Quartz Cron `0 0 /2 * * ?`（每 2 小时）
- 执行顺序：
  1. `SyncDepartments()`
     - 取本地最后时间（`SJ.Hrm_Inf_Department` 的 `CreatTime` 最大值）
     - 调 `DepartmentApi/GetDepartmentAllInfo?modifyTime=yyyy-MM-dd`
     - 批量写入 `SJ.Hrm_Inf_Department`
     - 执行 `SJ.Hrm_Pro_Department`
  2. `SyncPersons()`
     - 取本地最后时间（`SJ.Hrm_Inf_Person` 的 `CreatTime` 最大值）
     - 调 `EmployeeApi/GetEmployeeAllInfo?ModifyDate=yyyy-MM-dd`
     - 批量写入 `SJ.Hrm_Inf_Person`
     - 执行 `SJ.Hrm_Pro_SyncPerson`

## 三、钉钉同步人员过程（当前库可确认版）

> 这里按你要的“钉钉同步人员过程”做清晰梳理。  
> 结论是：当前库里人员同步主体是 `Sync_EmployeeToOA` / `Sync_EmployeeToDingTalk` 这类链路；本次我们已拿到并解读 `Sync_EmployeeToOA`（可直接反映 HR 人员同步模型）。

### 1) 过程：`dbo.Sync_EmployeeToOA`
- 同步范围：
  - 部门（`t_base_department` -> OA 中间表）
  - 岗位（`v_Sync_OA_Post` -> OA 中间表）
  - 人员（`t_base_employee + t_base_department + v_Sync_OA_Post` -> OA 中间表）
- 增量依据：
  - `t_base_employee.ModifyTime > sys_DataSyncRecord(ID=3).LastDate`
- 输出目标（OA 中间表）：
  - `MID_SJ_HRMDEPARTMENT`
  - `MID_SJ_HRMJOBTITLES`
  - `MID_SJ_HRMRESOURCE`
- 同步完成后更新时间戳：
  - `sys_DataSyncRecord`（`ID = 3`）

### 2) 数据表角色（按链路）
- HRM 正式数据源（本库）：
  - `dbo.t_base_department`
  - `dbo.t_base_employee`
  - `dbo.t_base_duty` / `dbo.v_Sync_OA_Post`
- 对外交换层（OA 中间表）：
  - `ECOLOGY...MID_SJ_HRMDEPARTMENT`
  - `ECOLOGY...MID_SJ_HRMJOBTITLES`
  - `ECOLOGY...MID_SJ_HRMRESOURCE`

## 四、字段级映射（API -> 中间层 -> 目标层）

> 说明：你要的是 `API 字段 -> 中间表字段 -> 目标表字段`。

以下为你提供的钉钉开放平台文档入口，用于**在钉钉侧创建/维护部门与用户**；与本文前面梳理的「HRM → 中间表 / OA」是**不同层级**：老系统里通常先有 HRM 主数据，再通过任务或接口把变更推到钉钉（部门表上有 `dingtalk_id` 等字段，具体以你们 `Sync_*DingTalk*` 作业与 `DingTalkAPI` 实现为准）。

### 与本文链路如何对齐（便于对照设计）

- **钉钉侧**：部门有钉钉 `dept_id`、用户有 `userid` 等；创建/更新需企业应用 `access_token` 及通讯录相关权限；详见上述文档中的请求参数与错误码。
- **HRM 侧（本文）**：`v_t_base_department` / `t_base_department` 含 `dingtalk_id` 等字段，用于与钉钉部门 ID 映射；人员同步增量常依赖 `ModifyTime`。
- **若要做「字段级映射：HRM → 钉钉 OpenAPI」**：需在文档中单独开表，把 HRM 的 `SJCode`、部门 GUID、`code`（工号）与钉钉的 `dept_id` / `userid` 的**生成与回填规则**写死（谁先创建、冲突谁优先、离职是否删用户等）。这部分依赖你们现有 `Sync_DepartmentToDingTalk` / `Sync_EmployeeToDingTalk` 代码与线上配置，未在本文展开。
