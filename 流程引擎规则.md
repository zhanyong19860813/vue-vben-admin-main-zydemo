# 流程引擎规则（冻结版）

> 目的：作为后续流程引擎开发的唯一执行规范。  
> 范围：审批节点、批次与会签、前序节点操作者、连线路由、手动干预、待办处理。  
> 原则：前端配置、后端解析、运维干预三方语义必须一致。

---

## 1. 术语与映射

### 1.1 会签术语映射

- 前端 `或签` = 泛微 `非会签` = 引擎 `any`
- 前端 `会签` = 泛微 `会签` = 引擎 `all`
- 前端 `依次` = 泛微 `依次逐个提交` = 引擎 `sequential`
- 前端 `抄送` = 引擎 `cc`（不参与主线通过判定）

### 1.2 关键数据对象

- `engineModel.nodes[]`：引擎节点定义
- `engineModel.edges[]`：出口路由定义（`priority` + 条件）
- `approvalOperatorRows[]`：审批节点操作者明细（执行主数据）
- `approvalHeaderConditionRules[]`：批次/条件结构化规则
- `manualIntervention.rules[]`：人工干预规则（强制 from -> to）

---

## 2. 冻结决策总览（已确认）

### 2.1 审批人解析优先级（对应 12.1）

- 执行顺序固定为：  
  `manualIntervention > approverResolve > assigneeStrategies > approvalWeaverUi(仅回显)`
- `approvalWeaverUi` 只用于界面回显与辅助生成，不直接抢执行优先级。

### 2.2 同批次通过/拒绝边界（对应 12.2）

- `all`：全员 `approve` 才通过；任一 `reject` 立即拒绝。
- `any`：任一 `approve` 立即通过；单个 `reject` 不立即拒绝，需“全员 reject”才拒绝。
- `sequential`：顺序审批；任一 `reject` 立即拒绝。

### 2.3 条件类型与比较（对应 12.3）

- 采用强类型配置：规则中必须显式声明 `formFieldType`。
- 操作符按类型收敛：
  - `boolean`：`==` / `!=`
  - `string`：`==` / `!=`
  - `number`：`== != > >= < <= between not_between`
  - `date`：`== != > >= < <= between not_between`
- 区间操作使用 `value` + `valueTo`。

### 2.4 前序节点操作者语义（对应 12.4）

- 多人通过时只取“第一位通过人”。
- 取值为“经理”时，取该第一位通过人的经理。
- 新增节点配置：`是否允许自动跳过`（默认 `true`）  
  - `true`：若前序解析人与当前节点办理人相同，自动跳过当前节点。  
  - `false`：即使同人也不跳过（适用于当前节点有必填表单场景）。

### 2.5 手动干预生效时机（对应 12.5）

- 采用“实时覆盖”：
  - 出当前节点路由前即时生效；
  - 命中 `fromNodeId -> forceToNodeId` 后立即改道；
  - 旧路径已生成待办实时关闭/作废，仅保留新路径有效待办。

## 3. 执行数据源与一致性约束

### 3.1 执行数据源

- 引擎运行只读取 `approvalOperatorRows[]` 作为审批操作者执行源。
- 表头（对象类型、对象、批次/条件、审批策略）只用于配置/生成明细，不作为运行时主判据。
- 表头与明细冲突时，以明细为准。

### 3.2 引擎模型建议

- 后端解析优先使用 `engineModel`，不依赖 `graphData` 的画布信息。
- 节点执行推荐从 `nodes[].approveConfig` 读取归一化数据（若存在）。

### 3.3 节点绑定表单（新增冻结约束）

- 标准写入路径：`engineModel.nodes[].properties.formBinding`。  
  结构：`{ formCode, formName, fieldRules? }`。
- `fieldRules` 键规则：
  - 主表字段：`fieldName`
  - 上表+下页签列：`tabKey::field`
- 运行与预览一律以 `formBinding` 为标准来源；旧字段仅兼容读取：
  - 名称：`formName | bindFormName | formTitle | formLabel`
  - 编码：`formCode | bindFormCode`
- 新增/编辑时禁止回写旧字段；旧字段仅用于历史数据回显与迁移期兜底。

## 4. 节点执行主流程

单节点推进必须按以下顺序：

1. 进入节点，检查是否命中手动干预（最高优先级）。
2. 解析审批操作者（基于 `approvalOperatorRows[]`）。
3. 按批次规则挑选 `activeBatch`（首个命中批次）。
4. 在 `activeBatch` 内按 `signMode` 聚合审批结果。
5. 节点通过后，按边规则选择下一节点。
6. 节点拒绝后，按驳回策略或结束策略处理。

---

## 5. 批次规则（强制）

### 5.1 基础

- `batch` 越小优先级越高。
- 批次条件为空视为恒真。

### 5.2 不同批次

- 按 `batch` 升序逐批判断。
- 找到第一个命中批次后立即停止查找（命中即停）。
- 未命中的后续批次本轮不参与。

### 5.3 相同批次

- 同批默认并发分发。
- 同批内实际聚合行为由每行 `signMode`（或会签属性）决定。

### 5.4 泛微对齐

- 不同批次顺序命中即停。
- 相同批次同时收到。
- 默认语义对齐“非会签”；如配置为会签/依次，以配置值执行。

---

## 6. 会签属性执行语义

### 6.1 `any`（或签/非会签）

- 同批任务并发发起。
- 任一 `approve`：节点立即通过。
- 其余未处理任务：自动关闭（`auto_closed`）。
- 只有“同批全员 `reject`”才判定节点拒绝。

### 6.2 `all`（会签）

- 同批任务并发发起。
- 全员 `approve` 才通过。
- 任一 `reject`：节点立即拒绝。

### 6.3 `sequential`（依次）

- 按配置顺序逐个激活任务。
- 当前通过才进入下一人。
- 任一 `reject`：节点立即拒绝。

### 6.4 `cc`（抄送）

- 不阻塞主流程推进。
- 仅影响抄送任务状态（需提交/不需提交）。

---

## 7. 条件规则与类型转换

### 7.1 结构化规则

- 优先使用结构化条件：`approvalHeaderConditionRules`、`edge.ruleGroups/rules`。
- 字符串条件 `condition` 仅兜底，不作为首选。

### 7.2 字段类型（两处配置，语义对齐）

**A. 审批表头条件（`approvalHeaderConditionRules` 中 `type=form_compare`）**

- 必须有 `formFieldType`：`string | number | boolean | date`
- `between/not_between` 必须提供 `value` 与 `valueTo`

**B. 连线出口条件（`WfEdgeConditionRule`，存于 `engineModel.edges[].ruleGroups` / `rules`）**

- 必须有 `fieldType`：`string | number | boolean | date`（设计器保存；**旧数据缺省按 `string`**）
- 操作符随 `fieldType` 收敛，原则与 **A** 一致：`boolean` 仅 `==`/`!=`，`string` 可用包含/列表类操作符，`number`/`date` 可用比较符及 `between`/`not_between`
- `between/not_between` 必须同时提供 `value`（左端/下限）与 `valueTo`（右端/上限）

### 7.3 比较约束

- `number`：执行前转 number，失败即规则非法。
- `boolean`：只允许 true/false。
- `date`：使用统一日期格式并在引擎内统一解析。
- `string`：不做隐式数值转换。

---

## 8. 前序节点操作者规则

### 8.1 取值范围

- 只允许引用“当前节点之前”的节点。

### 8.2 解析语义

- 前序多人通过时，仅取第一位通过人。
- 取“经理”时，取该第一位通过人的经理。

### 8.3 自动跳过

- 配置项：`approvalWeaverUi.autoSkipWhenSameActor`（默认 `true`；旧字段 `nodeOperatorAutoSkip` 仅作兼容读取）。
- 同人命中且为 `true`：当前节点自动跳过。
- 为 `false`：不跳过，仍创建待办并进入办理。

---

## 9. 手动干预规则

### 9.1 数据结构

- `manualIntervention.enabled`
- `manualIntervention.rules[].{ id, enabled, fromNodeId, forceToNodeId, reason }`

### 9.2 生效时机

- 出当前节点路由前即时判定。
- 命中后直接跳到 `forceToNodeId`，不再评估原路由。

### 9.3 待办处理

- 原路径已创建且失效的待办需实时关闭/作废。
- 新路径按目标节点重建有效待办。
- 必须记录审计日志（操作人、时间、原因、旧路径、新路径）。

---

## 10. 连线路由规则

节点通过后，连线判定顺序固定：

1. `priority` 升序
2. `ruleGroups`（组间 OR，组内 AND）
3. `rules`（AND）
4. `condition`（字符串兜底）
5. 命中第一条即路由；未命中走默认出口或抛配置异常

结构化规则中每条条件为 `WfEdgeConditionRule`：须按 **§7.2-B** 使用 `fieldType` 与 `valueTo`（区间），引擎按类型求值后再判定是否命中该边。

---

## 11. 保存与发布校验

### 11.1 图结构校验

- 仅一个开始节点
- 至少一个结束节点
- 无孤立审批节点

### 11.2 节点校验

- 审批节点必须可解析出至少一个有效操作者
- `approvalOperatorRows[]` 每行至少包含：`kind`, `name`, `batch`
- 区间操作必须具备 `valueTo`
- 节点绑定表单校验：
  - `formBinding` 存在时，至少有 `formCode` 或 `formName` 之一
  - `fieldRules` 取值必须为：`visible | hidden | readonly`
  - 键满足主表 `fieldName` 或页签列 `tabKey::field`

### 11.3 批次校验

- 同批建议单一 `signMode`
- 混用时必须有明确策略，否则阻止保存

### 11.4 路由校验

- 同源边 `priority` 不建议重复
- 条件表达式解析失败阻止保存
- 结构化连线条件：`fieldType` 与 `op` 组合合法；`between`/`not_between` 必须带 `valueTo`

### 11.5 干预校验

- `fromNodeId`、`forceToNodeId` 必须存在
- 禁止 `fromNodeId == forceToNodeId`

### 11.6 流程管理本地草稿与回灌（前端约束）

- 流程管理页使用 `wf-process-mgmt-local-draft:<processKey>` 存本地草稿。
- 节点绑定保存后、流程菜单切换前，前端应静默自动持久化草稿。
- 切流程加载草稿后，若画布未就绪，允许短时重试回灌（防止时序导致 `apply` 失败）。
- 草稿回灌成功后，节点信息表应与画布实时同步（避免“草稿有绑定、界面显示未绑定”）。

### 11.7 预览数据来源（前端约束）

- 点击「预览」时优先读取当前草稿快照中的开始节点 `formBinding`，按 `formCode` 拉取表单 `schemaJson` 动态渲染。
- 新标签页预览数据需支持跨页共享（至少 `localStorage` 兜底），避免仅 `sessionStorage` 导致回退静态模板。

## 12. 引擎伪代码（可直接转实现）

```text
advanceNode(instance, nodeId):
  node = getNode(nodeId)

  manual = matchManualIntervention(instance, nodeId)
  if manual.hit:
    closeInvalidTodos(instance, fromNodeId=nodeId)
    auditManualJump(instance, manual)
    return gotoNode(instance, manual.forceToNodeId)

  rows = loadOperatorRows(node)  // approvalOperatorRows or approveConfig.rows
  activeBatch = pickFirstMatchedBatch(rows, instance.formData, instance.actorContext)
  if activeBatch == null:
    return fail("NO_MATCHED_BATCH")

  result = runBatch(activeBatch) // any/all/sequential/cc
  if result == REJECTED:
    return applyRejectPolicy(instance, node)
  if result != APPROVED:
    return wait()

  next = pickEdgeByPriority(nodeId, instance) // ruleGroups > rules > condition
  if next == null:
    return handleNoRoute(instance, nodeId)
  return gotoNode(instance, next.targetNodeId)
```

## 13. 回归用例最小集

1. 不同批次命中即停（批次 0 命中后不再看批次 1）
2. 同批 `any`：一人通过即过；全拒绝才拒
3. 同批 `all`：一票拒绝即拒
4. `sequential`：按顺序推进，拒绝即终止
5. `node_operator`：多人通过时取第一位通过人
6. `autoSkipWhenSameActor=true`：同人自动跳过
7. `autoSkipWhenSameActor=false`：同人不跳过
8. 手动干预实时覆盖：立即改道 + 旧待办实时关闭
9. 表头与连线：`number`/`date` 的 `between`/`not_between` 区间比较（含非法值拦截）
10. 路由优先级：`ruleGroups > rules > condition`
11. 连线 `WfEdgeConditionRule`：`fieldType` 与操作符收敛、旧数据缺省 `string` 兼容
